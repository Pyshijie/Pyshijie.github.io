BIO,NIO到IO多路复用的演变过程

|字数总计:8.1k|阅读时长:26分钟|阅读量:

Redis之高性能设计epoll和IO多路复用深度解析

本文采用一个总分总的形式，来说明从BIO到NIO到IO多路复用的演进过程，以Redis为例来进行说明

Redis架构图解析

我们都知道Redis是单线程的，通过单线程可以处理多并发客户端请求，但是为什么可以通过单线程实现高性能呢？

我首先给IO多路复用一个解释：

Redis 是跑在单线程中的，所有的操作都是按照顺序线性执行的，但是由于读写操作等待用户输入或输出都是阻塞的，所以 I/O 操作在一般情况下往往不能直接返回，这会导致某一文件的 I/O 阻塞导致整个进程无法对其它客户提供服务，而 I/O 多路复用就是为了解决这个问题而出现

所谓 I/O 多路复用机制，就是说通过一种机制，可以监视多个描述符，一旦某个描述符就绪（一般是读就绪或写就绪），能够通知程序进行相应的读写操作。这种机制的使用需要 select 、 poll 、 epoll 来配合。多个连接共用一个阻塞对象，应用程序只需要在一个阻塞对象上等待，无需阻塞等待所有连接。当某条连接有新的数据可以处理时，操作系统通知应用程序，线程从阻塞状态返回，开始进行业务处理

Redis 服务采用 Reactor 的方式来实现文件事件处理器（每一个网络连接其实都对应一个文件描述符）
Redis基于Reactor模式开发了网络事件处理器，这个处理器被称为文件事件处理器。它的组成结构为4部分：
多个套接字、
IO多路复用程序、
文件事件分派器、
事件处理器。
因为文件事件分派器队列的消费是单线程的，所以Redis才叫单线程模型

接下来我们先来看一下Redis的整体架构：

解释一下上图的含义：

首先有多个客户端，假设有三个客户端A,B,C，客户端A想要建立连接，客户端B想要输入命令“set key value”，客户端C想要获取key的数据 “get key”，每个客户端与Redis之间想要交互，都需要建立一个socket连接，通过socket连接来进行交互，Redis采用IO多路复用机制，IO多路复用（假设是某个东西，机器人），客户端发送请求建立连接后，机器人就会将每个客户端的socket保存起来，然后机器人监听每个客户端的请求，监听到请求后，将请求事件加入到事件队列（按照先来后到的顺序），接着事件派发器从事件队列中取出事件，将其交给相应的事件处理器。

由于事件派发器从事件队列中取事件是单线程的，因此Redis是单线程的。

IO多路复用的含义

I/O:指的就是网络I/O

多路： 指多个客户端连接

复用：指复用一个或者几个线程，也就是说一个或者一组线程处理多个TCP连接，使用单进程就能够实现同时处理多个客户端的连接

其发展阶段可以分为：select -> poll -> epoll

基础概念

同步

同步指的是，调用者发出请求后，要一直等待服务者结果的通知后才能进行后续的执行

异步

异步指的是，调用者发出请求，服务者接收到请求后，先返回应答让调用者先回去，然后服务者再计算结果，计算出结果后，会通知并返回给调用者

阻塞

阻塞指的是，调用方发出请求后，一直等待着结果什么事情也不做，当前线程/进程会被挂起

非阻塞

非阻塞指的是，调用方发出请求后，就去忙别的事情了，不会挂起当前线程/进程，而会立即返回

上述都是官方术语，显得比较难以理解，接下来我就生活中的例子来将这四种构成的组合演示一遍

Blocking IO - 阻塞IO

阻塞IO指的是：当用户进程调用了recvfrom这个系统调用，kernel就开始了IO的第一个阶段：准备数据（对于网络IO来说，很多时候数据在一开始还没有到达。比如，还没有收到一个完整的UDP包。这个时候kernel就要等待足够的数据到来）。这个过程需要等待，也就是说数据被拷贝到操作系统内核的缓冲区中是需要一个过程的。而在用户进程这边，整个进程会被阻塞（当然，是进程自己选择的阻塞）。当kernel一直等到数据准备好了，它就会将数据从kernel中拷贝到用户内存，然后kernel返回结果，用户进程才解除block的状态，重新运行起来。所以，BIO的特点就是在IO执行的两个阶段（第一个阶段是等待数据阶段，第二个阶段是数据从内核复制到用户空间阶段）都被block了。

注意：这两个阶段都是阻塞状态，阻塞（意味着这个进程在这一段时间内什么都不能干）

从上述流程我们可以看出阻塞IO，从开始到结尾都是阻塞的，阻塞的时候就什么都不干，于是当客户端1一直开着不发送数据的时候，后续来再多的客户端，服务端都不会管，就造成了极大的资源浪费，性能也很低。

通过分析，我们发现了上述的问题，怎么解决呢？

解铃还须系铃人，从根本出发是什么导致了这样的情况出现，是因为我们的accept()和read()都是阻塞的，最关键的是，只有一个主线程在阻塞监听或者读取，如果我们每连接一个客户端就开辟一个线程出来，那么就可以解决上述的问题了。每个线程都是一个ServerSocket和Socket的交互，互不干扰，这样就是这样的架构。

可以发现使用多线程的方式，解决了阻塞IO的问题，那么多线程的方式又有什么缺点呢？

多线程的方式，每新来一个客户端都要开启一个新的线程来维护它与服务端之间的交互。

如果来了大量的客户端，那么很容易造成线程资源的浪费，而且线程的创建，在操作系统的用户态中不能直接开辟，需要调用内核来完成线程的创建，用户态到内核态涉及到中断机制等，耗费资源，并且多线程的情况下，线程上下文切换带来的开销也不容小觑，十分耗费资源。

解决方案:

由于线程数量多导致了这个问题，那么我们可以采用线程池的方案来解决，但是当线程很多的时候，池的大小并不能很好的进行控制，少量客户端可以使用线程池的方式。

第二种解决方案，我们从源头出发是什么导致了我们需要创建多个线程，正是因为read()和accept()的阻塞，因此我们可以尝试让这两个操作成为非阻塞模式，于是就引出了下文的非阻塞IO。

NoneBlocking IO - 非阻塞IO

非阻塞IO的具体流程如下：

当用户进程发出read操作时，如果kernel中的数据还没有准备好，那么它并不会block用户进程，而是立刻返回一个error。从用户进程角度讲 ，它发起一个read操作后，并不需要等待，而是马上就得到了一个结果。用户进程判断结果是一个error时，它就知道数据还没有准备好，于是它可以再次发送read操作。一旦kernel中的数据准备好了，并且又再次收到了用户进程的system call，那么它马上就将数据拷贝到了用户内存，然后返回。所以，NIO特点是用户进程需要不断的主动询问内核数据是否准备好了，没有准备好就再次询问，直到准备好才进行数据的读取。

非阻塞IO，两个操作都是非阻塞的。

  • accept()操作是非阻塞的，当没有客户端连接的时候，返回error，有的话返回一个socket。
  • read() 操作是非阻塞的，当读取不到数据的时候，返回error，读取到的话就进行数据的读取。

在非阻塞式 I/O 模型中，应用程序把一个套接口设置为非阻塞，就是告诉内核，当所请求的I/O操作无法完成时，不要将进程阻塞而是返回一个“错误”，这样进程就知道什么情况了，应用程序基于 I/O 操作函数将不断的轮询数据是否已经准备好，如果没有准备好，继续轮询，直到数据准备好为止。

代码实现，仍然是一个服务端，两个客户端，此时由于accept()和read()操作都是非阻塞的，每一个客户端来连接后，accept()就返回一个socket之后，那么这个accpet操作已经完成了，因此我们想要保证服务端与客户端之间的通信，需要将socket进行一个保存，保存在服务端，服务端可以有一个socket列表，每个连接成功的客户端都要保存在这个列表中，之后服务端通过不断轮询这个列表，看看每一个socket是否有可读数据，如果有进行读取，如果没有继续下一个。那么整体的架构可以是这样的。

  • 第三：还有一个缺点就是，我们每次都要遍历一整个socket集合，但是可以会出现，十万个socket中，只有2个socket有数据可以读取，但是我们仍然白白遍历了99998个socket，而且还调用了内核的read（），造成了大量资源浪费，并且造成了CPU空转，消耗大量资源。

解决方案：我们最严重的缺点就是，一直在进行用户态和内核态之间的切换，我们可以将socketList传送到内核中，让内核帮我们完成遍历，并且在遍历过程中，由于是在内核中，read()函数的调用也不会涉及到用户态到内核态的切换。

于是IO多路复用模型就诞生了，从select -> poll -> epoll

IO multiplexing - IO多路复用

在介绍IO多路复用之前，先介绍一些专用术语：

fd:文件描述符

文件描述符（File descriptor）是计算机科学中的一个术语，是一个用于表述指向文件的引用的抽象化概念。文件描述符在形式上是一个非负整数。实际上，它是一个索引值，指向内核为每一个进程所维护的该进程打开文件的记录表。当程序打开一个现有文件或者创建一个新文件时，内核向进程返回一个文件描述符。在程序设计中，一些涉及底层的程序编写往往会围绕着文件描述符展开。但是文件描述符这一概念往往只适用于UNIX、Linux这样的操作系统。

Reactor设计模式

Reactor 模式，是指通过一个或多个输入同时传递给服务处理器的服务请求的事件驱动处理模式。服务端程序处理传入多路请求，并将它们同步分派给请求对应的处理线程，Reactor 模式也叫 Dispatcher 模式。即 I/O 多了复用统一监听事件，收到事件后分发(Dispatch 给某进程)，是编写高性能网络服务器的必备技术。

Reactor 模式中有 2 个关键组成：
1）Reactor：Reactor 在一个单独的线程中运行，负责监听和分发事件，分发给适当的处理程序来对 IO 事件做出反应。 它就像公司的电话接线员，它接听来自客户的电话并将线路转移到适当的联系人；
2）Handlers：处理程序执行 I/O 事件要完成的实际事件，类似于客户想要与之交谈的公司中的实际办理人。Reactor 通过调度适当的处理程序来响应 I/O 事件，处理程序执行非阻塞操作

Redis 服务采用 Reactor 的方式来实现文件事件处理器（每一个网络连接其实都对应一个文件描述符）,因此Redis是单线程的

select 是 I/O多路复用的一种实现

select 函数监视的文件描述符分3类，分别是readfds、writefds和exceptfds，将用户传入的数组拷贝到内核空间

调用后select函数就会阻塞，直到有文件描述符就绪（有数据 可读、可写、或者有except）或超时（timeout指定等待时间，如果立即返回设为null即可），函数返回。

其中&rset是一个bitmap结构，每一个文件描述符对应bitmap中的一位，这样就能找到对应关系，之后调用select函数，select函数的具体执行流程就是，我们上述的NIO搬到了内核中来操作，首先遍历fd文件描述符数组，判断哪个fd可读或者可写了，在bitmap中找到文件描述符对应的位，将其置为1，此时select就可以进行返回了，不用阻塞了。

然后由于只有5个客户端连接，那么肯定就是5个文件描述符，依次遍历判断bitmap中对应的位是否被置为了1，如果置为了1那么表示该socket已经准备好了，可以进行操作了。

select的出现在内核层帮助我们完成了socket就绪的统计，用户层的我们只需要遍历&rset，看看哪一位被置位为1了，那么就知道可以对那个socket进行操作了，不需要每次调用系统调用来判断了，效率有很大的提升。

select 其实就是把NIO中用户态要遍历的fd数组(我们的每一个socket链接，安装进ArrayList里面的那个)拷贝到了内核态，让内核态来遍历，因为用户态判断socket是否有数据还是要调用内核态的，所有拷贝到内核态后，这样遍历判断的时候就不用一直用户态和内核态频繁切换了从代码中可以看出，select系统调用后，返回了一个置位后的&rset，这样用户态只需进行很简单的二进制比较，就能很快知道哪些socket需要read数据，有效提高了效率

但是select有什么缺点呢？

  • 1.由于select底层使用到lbitmap，而bitmap最大1024位，也就是说一个进程最多只能处理1024个客户端
  • 2.&rset不可重用，每次遍历结束后都要重新创建bitmap下次使用新创建好的，如果重用的话那么原先被置为1的描述符可能没有准备好，再次系统调用就会产生问题
  • 3.文件描述符数组拷贝到了内核态(只不过无系统调用切换上下文的开销。（内核层可优化为异步事件通知）)，但是仍然有开销。select 调用需要传入 fd 数组，需要将数据拷贝一份到内核，高并发场景下这样的拷贝消耗的资源是惊人的。（可优化为不复制）
  • 4.select并没有通知用户态哪一个socket有数据，仍然需要O(n)的遍历。select 仅仅返回可读文件描述符的个数，具体哪个可读还是要用户自己遍历。（可优化为只返回给用户就绪的文件描述符，无需用户做无效的遍历）

poll

​

fd ：表示文件描述符

events：表示请求的事件类型，读或者写

revents：如果该文件描述符有事件发生就置位

讲解一下上述代码发生的流程：

首先将5个fd从用户态拷贝到了内核态中，然后遍历了5个客户端连接，将每个描述符的events置为了POLLIN，接着调用poll方法，poll方法为阻塞方法，执行poll方法（也会阻塞等待有一个或多个描述符变为就绪），如果文件描述符中有数据，那么会将fd对应的revents置为POLLIN，之后poll方法返回

循环遍历文件描述符，判断那个描述符的revents被置为了POLLIN，找到后将revents重新置为0，便于复用，之后对置位的描述符进行读取和处理

poll方法与select方法差不多，只是poll定义了自己的结构体，pollfd，使用数组存放了pollfd，通过revents标志来进行置位

poll使用pollfd数组来代替select中的bitmap，数组没有1024的限制，可以一次管理更多的client。它和 select 的主要区别就是，去掉了 select 只能监听 1024 个文件描述符的限制。

当pollfds数组中有事件发生，相应的revents置位为1，遍历的时候又置位回零，实现了pollfd数组的重用

poll解决了select的前两个缺点，但是后两个缺点还没有解决，于是引出了epoll函数

epoll

epoll 函数由三个函数组成，epoll_create和epoll_ctl,以及epoll_wait

epoll_create:创建一个epoll句柄

epoll_ctl:向内核中添加，删除，修改要监控的文件描述符

epoll_wait:类似select的调用

epoll的执行流程是这样的：（注意，**epoll是非阻塞的**）

1.当socket有数据的时候，会把相应的文件描述符进行"置位"（这里的置位不是真正的置位，因为epollfd没有revents标志），这里的置位是指将就绪的文件描述符放在一个就绪链表中。

2.然后epoll会返回有数据的文件描述符的个数

3.根据返回的个数nfds，读取就绪链表中的前N个文件描述符即可

4.进行读取，处理

epoll_create（）函数，会在内核开辟出一块内存空间，用来存放epoll中的fd的数据结构，这也就解决了频繁从用户态将一批文件描述符拷贝到内核空间的问题，此时只有第一次epoll时，需要拷贝，后续都不用拷贝了，因为epoll_create开辟的空间中已经存储了要监听的文件描述符

epoll_ctl() 函数，就是充当一个操作文件描述符的角色，它就是将每个socket的fd数据结构存放在epoll_create（）开辟的内存空间中

epoll_wait（）是一个阻塞函数（但是会阻塞一段时间，不是一直阻塞），当epoll_crate()中存放的文件描述符就绪后，就会将这些文件描述符fd放入到就绪链表中，然后返回就绪的fd个数

这样在用户态进行遍历的时候，就能明确知道那些文件描述符是就绪的了，因为就绪的文件描述符都存放在了就绪链表中，于是遍历就绪链表前N位即可。这样就解决了O(N)的时间复杂度问题，现在时间复杂度是O(1)了

结论

多路复用快的原因在于，操作系统提供了这样的系统调用，使得原来的 while 循环里多次系统调用，变成了一次系统调用 + 内核层遍历这些文件描述符。
epoll是现在最先进的IO多路复用器，Redis、Nginx，linux中的Java NIO都使用的是epoll。
这里“多路”指的是多个网络连接，“复用”指的是复用同一个线程。
1、一个socket的生命周期中只有一次从用户态拷贝到内核态的过程，开销小
2、使用event事件通知机制，每次socket中有数据会主动通知内核，并加入到就绪链表中，不需要遍历所有的socket

在多路复用IO模型中，会有一个内核线程不断地去轮询多个 socket 的状态，只有当真正读写事件发送时，才真正调用实际的IO读写操作。因为在多路复用IO模型中，只需要使用一个线程就可以管理多个socket，系统不需要建立新的进程或者线程，也不必维护这些线程和进程，并且只有真正有读写事件进行时，才会使用IO资源，所以它大大减少了资源的占用。多路I/O复用模型是利用 select、poll、epoll 可以同时监察多个流的 I/O 事件的能力，在空闲的时候，会把当前线程阻塞掉，当有一个或多个流有 I/O 事件时，就从阻塞态中唤醒，于是程序就会轮询一遍所有的流（epoll 是只轮询那些真正发出了事件的流），并且只依次顺序的处理就绪的流，这种做法就避免了大量的无用操作。 采用多路 I/O 复用技术可以让单个线程高效的处理多个连接请求（尽量减少网络 IO 的时间消耗），且 Redis 在内存中操作数据的速度非常快，也就是说内存内的操作不会成为影响Redis性能的瓶颈

三个方法的对比

既然epoll那么厉害了，为什么还要保留着select以及poll？

首先我们来看上面这张图，具体含义是，Redis对IO多路复用函数的选择：

在Solaries 操作系统上，采用的euport函数，在Linux操作系统上，采用的是epoll函数，在OSX,FressBSD上，采用的是kqueue函数，对于不是这三种操作系统的，采用的是select函数。作为备选方案。
